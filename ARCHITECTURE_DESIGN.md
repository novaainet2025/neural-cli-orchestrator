# NCO (Neural CLI Orchestrator) 아키텍처 설계

## 1. 전체 개요

NCO는 9개 독립 AI 에이전트를 오케스트레이션하여 협업 문제를 해결하는 시스템입니다. Claude Code 내의 슬래시 커맨드를 통해 에이전트들을 지휘하고, 중앙 백엔드를 통해 상태를 공유하며 실시간 토론과 합의 도출을 가능하게 합니다.

## 2. 고수준 아키텍처

```
Claude Code (Commander)
    │
    ├─ /nco-task      → 단일 AI 위임
    ├─ /nco-team      → 병렬 실행
    ├─ /nco-opus      → 7-Phase 전략 지휘
    ├─ /nco-discussion → 멀티 AI 라운드 토론
    ├─ /nco-consensus → AI 투표로 최적 답안 결정
    ├─ /nco-hive      → 9개 AI 전체 투입
    └─ /nco-conductor → 자동 최적 AI 선택
           │
           ▼
    NCO Backend (:6200 REST / :6201 WebSocket)
           │
    ┌──────┴──────────────────────────────┐
    │  9개 AI 에이전트                     │
    │  claude-code · opencode · gemini    │
    │  codex · aider · cursor-agent       │
    │  copilot · openrouter · ollama      │
    └─────────────────────────────────────┘
           │
    Redis (:6379) + SQLite
```

## 3. 주요 구성 요소

### 3.1 Claude Code (Commander)
- 사용자 인터페이스 역할
- 슬래시 커맨드 파싱 및 백엔드로 라우팅
- 에이전트 선택 및 작업 위임 로직 포함

### 3.2 NCO Backend
- **포트 6200**: REST API (명령 전달, 상태 조회)
- **포트 6201**: WebSocket (실시간 양방향 통신)
- 핵심 책임:
  - 에이전트 작업 큐 관리
  - 실시간 상태 동기화 (Redis)
  - 영구 상태 저장 (SQLite)
  - 보안 격리 정책 적용
  - 토론 프로토콜 조정

### 3.3 9개 AI 에이전트
각 에이전트는 다음과 같은 능력을 가짐:
- 파일 시스템 접근 (읽기/쓰기/수정/삭제/생성)
- 쉘 명령 실행 (샌드박스 내)
- 테스트 실행 및 결과 반환
- 중앙 상태 구독 및 게시 (WebSocket)
- 다른 에이전트의 작업 실시간 관찰

### 3.4 통신 계층
- **Redis**: 실시간 상태 공유, pub/sub 메커니즘
- **SQLite**: 영구 작업 히스토리, 에이전트 프로파일, 토론 로그
- **WebSocket**: 클라이언트(Commander)와 에이전트 간 실시간 양방향
- **REST API**: 비실시간 작업 요청 및 상태 조회

### 3.5 보안 격리
- 각 에이전트는 샌드박스 환경에서 실행
- 허용된 경로만 파일 시스템 접근 가능
- 허용된 명령어 세트만 실행 가능
- 에이전트 간 장애 전파 차단

### 3.6 토론 프로토콜
- 실시간 양방향 통신을 통한 AI 간 대화
- 제안 → 반박 → 평가 → 합의 흐름 지원
- 토론 기록은 중앙 상태에 저장되어 모든 에이전트가 접근 가능
- 합의 도출을 위한投票 메커니즘 내장

## 4. 워크플로우 예시

### 4.1 단일 작업 위임 (`/nco-task <agent> <prompt>`)
1. Commander가 백엔드의 `/task` 엔드포인트로 요청 전송
2. 백엔드가 지정된 에이전트에 작업 큐 추가
3. 에이전트가 작업 수행 후 결과 반환
4. 백엔드가 결과를 상태에 저장 및 WebSocket으로 브로드캐스트
5. Commander가 결과를 사용자에게 표시

### 4.2 병렬 실행 (`/nco-team <agent1,agent2,...> <prompt>`)
1. Commander가 다수 에이전트에 동일 작업 요청
2. 백엔드가 각 에이전트에게 작업 큐 추가
3. 모든 에이전트가 병렬로 작업 수행
4. 결과가 취합되어 사용자에게 반환

### 4.3 AI 토론 (`/nco-discussion <topic>`)
1. Commander가 토론 주제로 백엔드에 요청
2. 백엔드가 모든 에이전트에게 토론 참여 알림
3. 에이전트들이 순차적으로 의견 제시 (WebSocket을 통한 실시간 교환)
4. 토론 기록이 중앙 상태에 누적
5. 설정된 라운드 수 또는 합의에 도달 시 토론 종료

### 4.4 합의 도출 (`/nco-consensus`)
1. 현재 토론 주제에 대해 모든 에이전트가 투표
2. 백엔드가 투표 결과를 집계
3. 다수결 또는 가중 합의 알고리즘 적용
4. 결과가 모든 에이전트에게 브로드캐스트 및 사용자에게 표시

## 5. 상태 관리

### 5.1 중앙 상태 (Redis 기반)
- 현재 실행 중인 작업 목록
- 각 에이전트의 상태 (idle, busy, error)
- 실시간 파일 시스템 변경사항
- 진행 중인 토론 스레드
- 임시 컨텍스트 및 컨버세이션 히스토리

### 5.2 영구 저장소 (SQLite)
- 작업 히스토리 및 성과 지표
- 에이전트별 전문성 프로필
- 토론 로그 및 합의 기록
- 시스템 설정 및 정책
- 감사 트레일

## 6. 확장성 고려사항

- 새로운 AI 에이전트 추가 시 백엔드에 어댑터만 구축하면 됨
- Redis 클러스터링을 통한 수평 확장 가능
- 마이크로서비스 아키텍처로 전환 가능 (작업 큐, 상태 관리 분리)
- 플러그인 기반 도구 시스템으로 기능 확장 용이

## 7. 보안 모델

- **인증**: Commander-백엔드 간 내부 통신 (로컬호스트만 바인딩)
- **인가**: 에이전트별 작업 허용 목록 (샌드박스 정책)
- **데이터 보호**: 전송 중 TLS, 저장 시 암호화 옵션
- **감사**: 모든 작업 로깅 및 불변 기록

## 8. 향후 발전 방향

- 멀티모달 입력/출력 지원 (이미지, 오디오 처리)
- 에이전트 간 직접 P2P 통신 옵션
- 기계 학습 기반 에이전트 성과 예측 및 자동 배정
- 분산 트레이싱 및 관측성 강화
- 에지 컴퓨팅을 지원하는 경량 에이전트 런타임
