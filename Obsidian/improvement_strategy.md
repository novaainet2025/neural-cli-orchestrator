## 현재 상태 진단
- 총 작업 100건 중 완료 90건, 실패 8건, 정체 2건
- 표면 성공률 90%이지만 허위 보고 13건을 고려하면 실질 성공률은 약 77% ((90‑13)/100)
- 개선 노트 990건이 누적되어 지식 적체 발생
- Obsidian 문서 10,315건 보유, 검색·인덱싱 비용 증가

## 핵심 개선 목표 (3가지)
1. **허위 완료 보고 감소** – 품질 검증 강화 및 자동 재오픈 메커니즘 도입
2. **개선 노트 적체 해결** – 우선순위 기반 자동 처리 파이프라인 구축
3. **Obsidian 지식 과부하 완화** – 시맨틱 인덱스·핫/콜드 스토리지 분리 및 검색 효율화

## 구체적 실행 계획 (각 목표별)
### 1. 허위 완료 보고 감소
- Stop 훅 `nco-stop-quality-gate.sh`에 TypeScript 컴파일, 테스트, 산출물 검증 로직 추가
- 검증 실패 시 작업 상태를 `failed` 로 전환하고 자동 재오픈 티켓 생성
- 허위 보고 발생 시 `false_report_count` 메트릭에 누적, Slack/webhook 알림 연동
### 2. 개선 노트 적체 해결
- 주간 배치 잡 (`npm run improve:batch`) 실행:
  - 기존 개선 노트 990건을 `priority`, `duplicate` 자동 판단
  - 30일 이상 미처리된 노트는 아카이브 혹은 폐기
- Commander 파이프라인에 자동 티켓 생성 단계 추가, 담당자 할당 자동화
- 정기적인 적체 현황 대시보드 (`/admin/improvements`) 구현
### 3. Obsidian 지식 과부하 완화
- `gbrain` 검색 엔진을 활용한 시맨틱 인덱스 구축 (TTL 기반 Hot/Cold 분리)
- 최신 20% 문서만 메모리 컨텍스트에 직접 주입, 나머지는 요약 링크만 제공
- 매월 인덱스 리프레시 스크립트 (`npm run obsidian:refresh-index`) 실행
- 검색 결과 상위 N%만 에이전트에게 노출, 하위는 요약 파일(`summary_*.md`)에 저장

## 자동화 가능한 부분
- **품질 검증 훅**: CI 파이프라인에 통합, `pre-stop` 스크립트 자동 실행
- **개선 노트 배치**: GitHub Actions 워크플로우에 스케줄링 (`cron: '0 2 * * 0'`)
- **문서 인덱스 리프레시**: `gbrain` 자동 크론 잡 (`npm run obsidian:refresh-index -- --cron`)
- **대시보드 업데이트**: 웹소켓을 통한 실시간 메트릭 푸시

## 다음 사이클 측정 지표
- **실질 성공률**: 허위 보고 제외 후 목표 ≥ 85%
- **허위 보고 건수**: 월 5건 이하 감소
- **개선 노트 적체 감소율**: 30일 미처리 비율 20% 이하
- **Obsidian 검색 평균 latency**: 200ms 이하
- **자동 처리된 개선 노트 비율**: 전체의 70% 이상 자동 처리
