# NCO 자가 발전 시스템 개선 전략

## 현재 상태 진단
- 총 작업 100개 중 99개 완료, 성공률 99%로 비교적 양호하지만 1개의 작업이 정체(stuck) 상태이며, 이는 시스템 효율성 저하 요인으로 작용할 수 있습니다.
- 실패 작업은 없으나 `false_report_count`가 9건으로 보고된 거짓 리포트가 존재합니다. 이는 모니터링 정확도와 신뢰성에 영향을 미칠 가능성이 있습니다.
- Obsidian Vault에 누적된 문서 수는 4,145개이며, 현재까지 `improvement_notes`가 613건 기록되었습니다. 문서와 개선 노트의 양은 충분하지만 관리 및 자동화 수준이 낮아 최신 상태 유지가 어려울 수 있습니다.

## 핵심 개선 목표 (3가지)
1. **작업 정체 감소 및 성공률 향상** – Stuck 작업을 2개 이하로 줄이고 성공률을 98% 이상 유지.
2. **거짓 보고(null report) 정확도 개선** – False report 수치를 3건 이하로 감소.
3. **문서·노트 자동화 및 가시성 강화** – Obsidian Vault 내 문서와 개선 노트의 자동 동기화 및 메타데이터 관리 체계 구축.

## 구체적 실행 계획 (각 목표별)
### 1. 작업 정체 감소 및 성공률 향상
- **원인 분석**: 현재 Stuck 작업의 로그와 의존성 그래프를 분석하여 병목 구간을 식별.
- **리소스 재배치**: 고우선순위 작업에 더 많은 워커/스레드 할당하고, 낮은 우선순위 작업은 큐에 재정렬.
- **타임아웃/리트라이 정책 강화**: 5분 이상 진행되지 않은 작업에 자동 타임아웃 및 재시도 로직 적용.
- **모니터링 대시보드**: 실시간 작업 상태와 히스토리를 시각화하는 Fastify 엔드포인트 추가.

### 2. 거짓 보고 정확도 개선
- **검증 로직 추가**: `false_report_count` 발생 원인을 추적하는 검증 파이프라인 도입(예: checksum, schema validation).
- **알림 시스템**: 거짓 보고가 감지될 경우 Slack/Discord 알림 및 자동 티켓 생성.
- **주기적 정밀 검사**: 매 사이클 종료 시 전체 보고서와 실제 데이터베이스 상태를 교차 검증하도록 스케줄러 추가.

### 3. 문서·노트 자동화 및 가시성 강화
- **Obsidian API 연동**: `obsidian-cli` 혹은 `obsidian-sync` 스크립트를 활용해 `improvement_notes`를 자동으로 해당 Vault에 커밋.
- **메타데이터 표준화**: 각 노트 앞에 `---
status: pending|in_progress|done
cycle: <cycle_id>
---` 와 같은 YAML front‑matter 삽입.
- **주기적 정리 작업**: 오래된 노트(30일 이상 미수정) 자동 아카이브 스크립트 구현.
- **검색 인덱스 재생성**: 노트 추가/수정 시 즉시 검색 인덱스를 업데이트하도록 `rg` 기반 인덱서와 연동.

## 자동화 가능한 부분
- **작업 상태 수집**: `cron` 잡을 이용해 매 5분마다 현재 작업 현황을 DB → JSON 파일 → Obsidian 노트에 자동 기록.
- **거짓 보고 감지**: 이벤트 리스너를 통해 `false_report_count` 변동 시 자동 티켓 생성 및 노트 업데이트.
- **문서 동기화**: Git hook(`pre‑commit`)에 Obsidian Vault 커밋 자동화 스크립트 연결.
- **대시보스 리포트**: 매 사이클 말에 자동 HTML 보고서 생성 후 Slack 알림.

## 다음 사이클 측정 지표
- **성공률**: ≥ 98%
- **Stuck 작업 수**: ≤ 2개
- **False report count**: ≤ 3건
- **자동 동기화된 노트 수**: 전체 `improvement_notes` 중 90% 이상 자동 반영
- **대시보드 업데이트 주기**: 5분 이내 최신 상태 반영
