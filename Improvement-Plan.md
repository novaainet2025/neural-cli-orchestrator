## 현재 상태 진단
- 전체 작업 **100건** 중 **95건** 완료, 성공률 **95%**.  
- 실패 **5건**, 정체 **0건** → 미완료 **5건** 모두 실패 상태.  
- 허위 완료 보고 **4건** → 실제 신뢰 성공률 **(95‑4)/100 = 91%**.  
- Obsidian 문서 **9,693개** 대비 개선 노트 **971개** (비율 ≈ **1:10**) → 계획·실행 불균형.

## 핵심 개선 목표 (3가지)
1. **KPI 신뢰성 강화** – 허위 완료 보고 제거, 완료 자동 검증 파이프라인 구축.  
2. **잔여 실패 최소화** – 실패 작업 자동 재시도·원인 분석 체계 도입.  
3. **문서·메트릭 정합성** – Obsidian 문서와 개선 노트 비율 개선, 메트릭 자동 연동.

## 구체적 실행 계획 (각 목표별)
### 1. KPI 신뢰성 강화
- 완료 시 자동 검증 스텝 추가 (DB/Redis 상태, 파일 존재 확인).  
- 검증 실패 시 `false_report_count` 증가 및 알림.  
- 검증 로깅을 Obsidian 개선 노트에 자동 기록.
### 2. 잔여 실패 최소화
- 실패 작업에 대한 재시도 정책 (exponential backoff, max 3회).  
- 실패 원인 로그 집계·분류 자동화 (코드, 외부 서비스, 리소스).  
- 주간 리포트에 상위 3가지 반복 실패 원인 표시.
### 3. 문서·메트릭 정합성
- 매 사이클 종료 시 `obsidian_docs`와 `improvement_notes` 비율 계산 자동화.  
- 비율 <1:8 일 경우 신규 개선 노트 자동 템플릿 생성.  
- 문서 자동 태깅 및 링크 연결 스크립트 구현.

## 자동화 가능한 부분
- **완료 검증**: `opencode` → `ollama` 검증 단계에 후크 추가.  
- **재시도 로직**: `src/core/taskRunner.ts`에 재시도 래퍼 구현.  
- **메트릭 연동**: `src/utils/metrics.ts`에 Obsidian API 호출 스케줄러 추가.  
- **주간 리포트**: `cron` 잡으로 `src/reports/weekly.ts` 생성.

## 다음 사이클 측정 지표
- **신뢰 성공률**: 허위 보고 제외 후 성공률 ≥ **93%**.  
- **잔여 실패 건수**: ≤ **2건**.  
- **문서 비율**: 개선 노트 / Obsidian 문서 ≥ **1:8**.  
- **자동 검증 커버리지**: 전체 완료 작업 중 **100%** 검증 적용.
