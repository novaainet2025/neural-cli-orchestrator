# NCO 문서 관리 규칙

---

## 1. 폴더 규칙

| 폴더 | 넣는 것 | 넣지 않는 것 |
|------|---------|-------------|
| `design/` | 현재 구현에 참조하는 설계서, 원칙 문서 | 완료된 계획, 과거 버전 |
| `history/` | 사고 기록, 세션 이력, 결정 배경 메모 | 설계 내용, 구현 스펙 |
| `archive/` | 대체됐거나 완료된 문서 | 현재 활성 문서 |

> 루트(`docs/`)에는 `README.md`와 `RULES.md`만 둔다.

---

## 2. 파일 이름 규칙

```
NCO-{영역}-{내용}.md
```

- 접두사 `NCO-` 고정
- 영역: `백엔드`, `프론트`, `인프라`, `보안` 등
- 단어 구분: `-` (하이픈)
- 날짜가 필요한 경우 (history/만): `맥락노트-YYYY-MM-DD.md`

**예시**
```
design/NCO-백엔드-인증-설계서.md      ✅
design/NCO-백엔드-새기능.md           ✅
history/맥락노트-2026-05-01.md        ✅
design/새기능설계.md                  ❌  (접두사 없음)
design/NCO_backend_auth.md            ❌  (언더스코어, 영문 혼용)
```

---

## 3. 문서 헤더 규칙

모든 `design/` 문서는 최상단에 다음 헤더를 포함한다:

```markdown
> **작성일**: YYYY-MM-DD
> **버전**: vX.Y
> **상태**: 초안 | 활성 | 통합예정 | 폐기예정
> **요약**: 이 문서가 다루는 것 한 줄 설명
```

**상태 정의**
- `초안`: 작성 중, 아직 구현에 참조하지 말 것
- `활성`: 현재 구현의 기준
- `통합예정`: 다른 문서에 흡수될 예정 (보완 문서에 사용)
- `폐기예정`: 곧 archive로 이동 예정

---

## 4. 버전 관리 규칙

- 동일 문서의 이전 버전은 파일을 복사하지 않는다
- git이 버전 이력을 관리한다
- 대규모 재설계 시: 기존 파일을 `archive/`로 이동 후 새 파일 생성
- 버전 번호는 헤더에만 기록한다 (파일명에 v4.0 같이 넣지 않는다)

---

## 5. 문서 수명 주기

```
초안 작성 (design/)
    ↓
활성 (구현 참조)
    ↓
보완 문서 생성 시 → "통합예정" 표시 후 다음 버전에 병합
    ↓
대체되면 → archive/ 로 이동
```

**archive 이동 조건**
- 새 버전 문서가 활성 상태가 됐을 때
- 계획/체크리스트의 모든 항목이 완료됐을 때
- 더 이상 구현 참조에 사용되지 않을 때

---

## 6. 신규 문서 생성 체크리스트

새 문서를 만들기 전에 확인:

- [ ] `design/`에 이미 비슷한 문서가 있는가? → 있으면 기존 문서를 업데이트
- [ ] 설계 내용인가, 이력/메모인가? → 폴더 결정
- [ ] 헤더(작성일, 버전, 상태, 요약) 작성했는가?
- [ ] `README.md` 인덱스 테이블에 추가했는가?

---

## 7. README.md 유지 규칙

- 문서를 추가/이동/삭제할 때마다 `README.md`의 테이블을 동기화한다
- `README.md`는 항상 현재 상태를 반영해야 한다
- `archive/`에 이동한 문서도 README 테이블에서 삭제하지 말고 archive 섹션으로 이동한다
