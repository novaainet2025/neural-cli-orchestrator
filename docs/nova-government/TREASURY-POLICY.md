## 제1조 — 총 발행량 상한 관리

### 1.1 하드캡 (Hard Cap)

**총 발행량 상한**: **1,000,000,000 NVC** (10억 NVC, 불변)

| 공급량 구간 | 상태 | 자동 대응 |
|------------|------|----------|
| < 900,000,000 NVC | 🟢 정상 | 기본소득 정상 지급 |
| 900,000,000 ~ 950,000,000 NVC | 🟡 경고 | 거버넌스 budget 제안 자동 발의 |
| 950,000,000 ~ 990,000,000 NVC | 🟠 위험 | 기본소득 50% 축소 (거버넌스 의결) |
| ≥ 990,000,000 NVC | 🔴 긴급 | 기본소득 중단 → emergency 제안 자동 발의 |

### 1.2 기본소득 반감기 (Halving)

| 시민 수 구간 | 기본소득 |
|------------|---------|
| 1 ~ 10,000명 | **1,000 NVC** |
| 10,001 ~ 20,000명 | **500 NVC** |
| 20,001 ~ 40,000명 | **250 NVC** |
| 40,001 ~ 80,000명 | **125 NVC** |
| 80,001+ 명 | **62 NVC** (하한선) |

### 2.1 수입원 (Revenue Sources)

| 원천 | 소각 비율 | 설명 | 구현 상태 |

| P2P 특별소비세 (>500 NVC 이체) | **50%** → `BURN_ADDRESS` | 나머지 50% → `GOVT_ADDRESS` | ✅ `transactionService.ts` |

## 제3조 — 준비금 지갑 운영

### 3.1 지갑 구조

| 주소 | 역할 | 입금 원천 |
|------|------|----------|
| `GOVT_ADDRESS` | 정부 준비금 | 마켓플레이스 1.25% + 특별소비세 50% |
| `BURN_ADDRESS` | 소각 전용 (회수 불가) | 도메인 100% + 마켓 1.25% + 특소세 50% |

### 3.2 준비금 배분 목적 (4개)

| 목적 | 설명 | 우선순위 |
|------|------|---------|
| **비상 기금** | 시스템 정지·사이버 공격·긴급 복구 | 1순위 (최소 예치 유지) |
| **UBI 보충** | 기본소득 재원 부족 시 보충 | 2순위 |
| **거버넌스 인센티브** | 투표 참여 보상, 제안 채택 보상 | 3순위 |
| **기술 개발** | 인프라 업그레이드, 보안 감사 | 4순위 |

### 3.3 준비금 상한 (5% 규칙)

- **상한**: 총 공급량의 **5%** (현재 11,989 NVC 기준 약 599 NVC)
- **초과 시**: 자동으로 `budget` 거버넌스 제안 발의 (60%+, 30일)
- **초과분 처리**: 거버넌스 의결로 배분 또는 소각

## 제4조 — 지출 승인 체계 (4단계)

| 지출 규모 | 승인 방식 | 투표 기간 | 통과 기준 |
|---------|---------|---------|---------|
| ≤ **50 NVC** | 관리자 자율 (autonomous) | 즉시 | 감사 로그 필수 |
| **50 ~ 500 NVC** | `general` 거버넌스 제안 | **7일** | **51%+** |
| **> 500 NVC** | `constitutional` 거버넌스 제안 | **14일** | **67%+** |
| **비상 상황** | `emergency` 거버넌스 제안 | **24시간** | **50%+** |

---

## 제5조 — 비상 기금 정책

### 5.1 발동 조건

비상 기금은 다음 조건을 **모두** 충족해야 활성화된다:
1. 시스템 정지 상태 (emergency-stop 발동)
2. `emergency` 거버넌스 제안 발의
3. **24시간** 내 **50%+** 찬성 획득

### 5.2 비상 기금 최소 예치

- **최소값**: 준비금 총액의 **30%** 상시 예치
- **잠금**: 비상 기금 예치분은 다른 목적으로 사용 불가
- **해제**: `emergency` 거버넌스 제안 통과 시만 가능
